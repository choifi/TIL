# 자바는 컴파일시 바이트코드로 바뀐 다음에 기계어가 된다. 왜 그렇게 만들었을까?
자바가 소스 코드를 바로 기계어로 바꾸지 않고, 중간 단계인 '바이트코드(Bytecode)를 거치는 이유는 '한 번 써서 어디서든 실행한다(Write Once, Run Anywhere)'는 철학 때문입니다.

만약 자바가 곧장 기계어로 번역되었다면 윈도우용, 맥용, 리눅스용, 프로그램을 각각 따로 컴파일해서 배포해야 했을 것입니다. 

1. 플랫폼 독립성 (어디서든 실행 가능)
기계어는 CPU 아키텍처(Intel, Apple Silicon 등)와 운영체제에 따라 모두 다릅니다. 하지만 '바이트코드'는 특정 하드웨어가 아니라 'JVM(Java Virtual Machine)'이라는 가상의 컴퓨터가 이해하는 공통 언어입니다. 

    * 컴파일 타임: 개발자가 쓴 .java 파일을 .class (바이트코드)로 변환합니다. 이 파일은 어떤 OS에서도 똑같습니다. 
    * 런타임: 각 운영체제에 설치된 JVM이 이 바이트코드를 읽어서 해당 기기 전용 기계어로 실시간 번역합니다. 

2. 보안과 안정성 
바이트 코드는 실행되기 전 JVM의 '클래스 로더'와 '바이트코드 검증기'를 거칩니다. 
    * 코드가 메모리를 불법적으로 침범하는지, 로컬 파일을 함부로 조작하는지 미리 검사할 수 있습니다. 
    * 중간 단계에서 필터링을 한 번 거치기 때문에, 기계어로 바로 실행되는 언어보다 훨씬 안전하게 프로그램을 돌릴 수 있습니다. 

3. JIT 컴파일러를 통한 최적화
"중간 단계를 거치면 느리지 않을까?"라는 걱정이 들 수 있는데, 이를 위해 JIT(Just-In-Time) 컴파일러가 존재합니다.
    * 자주 실행되는 코드(Hotspot)를 찾아내어 실시간으로 아주 효율적인 기계어로 미리 바꿔둡니다.
    * 덕분에 처음에는 인터프리터 방식으로 천천히 실행되다가, 프로그램이 돌아갈수록 점점 속도가 빨라집니다. 

# 기계어란?
기계어(Machine Code)는 컴퓨처의 두뇌인 CPU가 즉각적으로 이해하고 실행할 수 있는 유일한 언어. 기계어는 철저하게 이진수(0과 1)로만 이루어져 있습니다. 

# JVM의 번역 방식: 인터프리터 vs JIT
JVM은 효율성을 위해 두 가지 방식을 섞어서 사용합니다. 
    * 인터프리터(Interpreter): 바이트코드를 한 줄씩 읽어서 즉시 기계어로 바꾸고 실행합니다. 한 줄씩 처리하니 실행은 빠르지만, 전체적인 속도는 느립니다. (반복문도 매번 새로 변역함)
    * JIT(Just-In-Time) 컴파일러: 프로그램이 돌아가는 도중(Runtime)에 "자주 호출되는 코드(HotSpot)"를 찾아냅니다. 이 부분을 통째로 기계어로 컴파일해서 '캐시에 저장'해둡니다. 다음에 같은 코드가 나오면 번역 없이 캐시에서 바로 실행하므로 속도가 획기적으로 빨라집니다. 

# C 컴파일러 vs 자바 컴파일러
두 언어는 '언제, 무엇을 위해 컴파일하는가'에서 큰 차이가 납니다.

                 C 언어 (AOT 방식)        |         자바 (JIT 방식)
컴파일 시점  |  실행 전 (빌드 타임)            |    실행 전(바이트코드) + 실행 중(기계어)
결과물      |  특정 OS용 실행 파일 (.exe 등)  |    플랫폼 독립적인 .class 파일
최적화      |  빌드 시점에 정적으로 최적화      |    실행 중 사용자 환경에 맞춰 동적 최적화
장점       |  초기 실행 속도가 매우 빠름       |    어떤 기기에서든 동일하게 작동함

# AOT (Ahead-Of-Time) 컴파일이란?
최근 자바에서도 많이 언급되는 AOT는 '미리(Ahead-Of-Time)" 기계어로 다 바꿔두는 방식입니다. 
    * 왜 쓸까? JIT는 실행 중에 번역하느라 초기 구동 시 CPU를 많이 쓰고 속도가 느립니다. (Cold Start 문제)
      AOT는 실행 전에 미리 기계어로 구워버리기 때문에 컴퓨터가 켜지자마자 최고 속도로 달릴 수 있습니다. 
    * 단점 : 대신 자바의 최대 장점은 "플랫폼 독립성"을 포기해야 합니다. 특정 OS 전용으로 파일이 만들어지기 때문입니다.
    * 활용 : 주로 서버리스(AWS Lambda)나 클라우드 네이티브 환경처럼 빠른 시작이 중요한 곳에서 GraalVM 같은 도구를 통해 사용됩니다. 

# 요약: JIT는 "달리면서 최적화", AOT는 "출발 전 완비"

# 메모리 관리의 자동화와 수동화, 그리고 OOM

# GC (Garbage Collection)란?
가비지 컬랙션은 JVM이 더 이상 사용되지 않는 객체(쓰레기)를 찾아내어 '자동으로 메모리에서 해제'해주는 프로세스입니다. 
    * 동작 원리 : "Reachability"라는 개념을 사용합니다. 어떤 객체에 접근할 수 있는 경로(Root)가 끊어지면, JVM은 해당 객체를 "쓰레기"로 간주하고 청소합니다. 
    * Stop-The-World : GC를 실행하기 위해 JVM이 애플리케이션 실행을 멈추는 현상입니다. 효율적인 GC 튜닝의 핵심은 이 멈추는 시간을 최소화하는데 있습니다. 

# OutOfMemory(OOM) 에러는 왜 뜰까?
이론적으로는 GC가 다 치워줘야 하지만, 감당 가능한 수준을 넘어서면 java.lang.OutOfMemoryError가 발생합니다. 주요 원인은 다음과 같습니다. 
    * 메모리 누수 (Memory Leak): 개발자가 실수로 객체 참조를 해제하지 않아 (예: Static 컬렉션에 계속 쌓아둠), GC가 "이건 아직 쓰는 거네?"라고 착각하여 지우지 못하는 경우입니다. 
    * 객체 폭주: 7천만 건의 데이터를 한꺼번에 리스트에 담으려 하는 것처럼, 힙(Heap) 메모리 크기보다 더 큰 데이터를 로드할 때 발생합니다. 
    * 부족한 설정: 서버 사양에 비해 -Xmx (최대 힙 크기) 설정값이 너무 작게 잡혀 있을 때도 발생합니다. 

# C언어(수동) vs 자바(자동) 메모리 관리 비교
C언어는 개발자가 malloc()으로 빌리고 free()로 직접 갚아야 합니다. 
          |     C언어 (Manual)              |       자바 (Automatic)
 제어권     |  개발자가 100% 통제               |   JVM이 관리 (비결정적)
 생산성     |  낮음 (실수가 많고 코드가 복잡해짐)    |  매우 높음 (비즈니스 로직에 집중)
 성능      |  오버헤드 없음 (가장 빠름)           | GC 실행 시 오버헤드 및 멈춤 발생
 안정성     | 낮음                            | 높음 (안전한 메모리 관리 모델)

 장단점 정리
    * C언어: 하드웨어 자원이 매우 제한적이거나 극강의 속도가 필요한 임베디드, OS 커널 등에 유리합니다. 하지만 사람이 실수하면 시스템 전체가 죽을 수 있습니다. 
    * 자바: 대규모 비즈니스 로직을 다룰 때 유리합니다. 메모리 해제 실수에서 자유로워지는 대신, JVM이 메모리를 관리하는 데 드는 비용(CPU, RAM 사용량)을 지불해야 합니다. 

# JVM의 힙(Heap) 메모리는 객체의 '생존 기간'에 따라 효율적으로 관리하기 위해 Young Generation과 Old Generation으로 나뉩니다. 

# 1. Young Generation: "금방 생성되고 사라지는 객체"
대부분의 객체는 생성된 후 금방 사용이 끝나고 쓰레기가 됩니다. 이 영역은 효율을 위해 3가지 세부 영역으로 나뉩니다. 
    * Eden (에덴): new 연산자로 생성된 따끈따끈한 객체들이 처음 머무는 곳. 이 곳이 꽉 차면 'Minor GC'가 발생
    * Survivor 0 & 1 (서바이버): Eden에서 GC가 일어날 때 살아남은 객체들이 이동하는 대피소입니다. 
        * 중요한 규칙: S0와 S1 중 '한 곳은 반드시 비어 있어야 합니다.' 두 영역을 번갈아 가며 객체를 옯기고, 살아남을 때마다 '나이(Age)'를 먹습니다. 

# 2. Old Generation: "오래 살아남은 끈질긴 객체"
Young 영역에서 정해진 '나이(Threshold)'를 다 채울 때까지 살아남은 객체들이 이동하는 곳입니다. (이 과정을 Promotion이라고 합니다.)
    * 보통 Young 영역보다 크게 할당됩니다.
    * 이 곳이 꽉 차면 'Major GC (또는 Full GC)'가 발생하며, 이때는 객체들을 검사하는 시간이 훨씬 오래 걸리고 시스템이 멈추는 Stop-The-World 현상도 더 길게 나타납니다.

# 3. GC가 일어나는 시나리오 (Step-by-Step)
1. 객체 탄생: 모든 객체는 우선 Eden에 쌓입니다. 
2. Minor GC 발생: Eden이 꽉 차면 GC가 돌아 살아남은 객체들만 'Survivor 0'으로 옮깁니다. (Eden은 비워짐)
3. 반복: 다시 Eden이 차면 GC가 돌아 Eden과 S0에 있는 객체 중 살아남은 것만 'Survivor 1'로 보냅니다. 
4. Promotion: 이 과정을 반복하며 끝까지 살아남은 '장수 객체'들은 'Old Generation'으로 승격됩니다. 
5. Major GC: Old 영역마저 꽉 차면 드디어 대청소인 Major GC가 실행됩니다. 


# 4. 왜 이렇게 복잡하게 나눌까?
만약 메모리를 나누지 않고 통째로 관리한다면, '새 객체 하나를 넣을 공간을 찾기 위해 메모리 전체를 다 뒤져야 할 수도 있습니다'
    * 효율성: 금방 죽을 객체(Young)는 빨리빨리 비워버리고, 오래 갈 객체(Old)는 따로 모아둠으로써 청소 횟수와 범위를 획기적으로 줄인 것입니다. 
    * 성능: 대부분의 Minor GC는 아주 빠르게 끝나기 때문에 사용자 입장에서 시스템이 멈췄다는 느낌을 거의 받지 않게 됩니다. 

백엔드 설계를 할 때, 리스트나 맵에 데이터를 너무 오래 담아두면 객체가 계속 Old 영역으로 넘어가 결국 성능 저하(Full GC)의 원인이 됩니다.

기존의 Java GC (Parallel GC나 CMS GC)가 힙을 'Young/Old로 딱딱하게 나누어 관리'했다면, G1(Garbage First) GC'는 이 고정관념을 완전히 깨버린 혁신적인 방법입니다. 

Java 9부터 기본(Default) GC로 채택된 이유이기도 하죠.

# 1. 리전 (Region)이란 무엇인가?
G1 GC는 힙 메모리를 고정된 두 덩어리(Young/Old)로 나누지 않습니다. 대신 전체 힙을 '동일한 크기의 작은 구역(Region)'들로 쪼갭니다.
    * 바둑판 형태: 힙을 약 2,048개의 구역으로 잘게 나눈다고 상상해 보세요.
    * 유연한 역할: 각 리전은 고정된 역할이 없습니다. 어떤 리전은 'Eden'이 되었다가, 청소가 끝나면 'Old'가 될 수도 있고, 'Survivor'가 될 수도 있습니다. 
    * Humongous 리전: 리전 크기의 50%를 넘는 거대한 객체만을 위한 특별한 구역도 존재합니다.

# 2. G1GC는 어떻게 작동하나? (Garbage First!)
이름이 왜 'G1 (Garbage First)일까요? 말 그대로 '쓰레기가 가장 많은 곳부터 먼저 치운다'라는 뜻

1. 스캐닝: 각 리전에 쓰레기가 얼마나 쌓였는지 실시간으로 파악합니다.
2. 순위 매기기: '이 리전은 90%가 쓰레기네?', '저 리전은 10%뿐이네?'를 판단하여 순위를 매깁니다. 
3. 선별적 청소: 전체를 다 치우는 게 아니라, 정해진 시간(Pause Time Goal) 내에 '쓰레기가 가장 많은 리전 위주로 골라서' 치웁니다. 

# 3. 이 방식의 장점 
1. Stop-The-World의 예측 가능성
기존 GC는 메모리가 커질수록 청소 시간도 기하급수적으로 늘어났습니다. 하지만 G1GC는 "사용자가 허용한 시간(예: 200ms) 내에 치울 수 있는 만큼만" 골라 치우기 때문에 시스템이 갑자기 길게 멈추는 현상을 방지할 수 있습니다. 

2. 효율적인 메모리 회수 
쓰레기가 꽉 찬 리전만 쏙쏙 골라 비우기 때문에, 적은 노력으로 많은 메모리를 빠르게 확보할 수 있습니다. 

3. 메모리 단편화(Fragmentation) 헤결
기존 방식은 객체를 지우고 나면 메모리 사이에 빈틈이 생기는 문제가 있었는데, G1GC는 청소 과정에서 살아남은 객체들은 '다른 리전으로 옮기며 차곡차곡 쌓아줍니다.(Compaction)' 덕분에 빈 공간이 깔끔하게 정리됩니다. 

4. 요약
    * 기존 GC: "거실(Young) 청소 끝났으니 이제 안방(Old) 전체를 대청소하자!" (시간 오래 걸림)
    * G1 GC: "집을 작은 구역으로 나누고, 지금 제일 더러운 구역 3군데만 10분 안에 빨리 치우자!" (효율적이고 빠름)

백엔드 시스템에서 '대용량 힙(4GB 이상)'을 사용하면서도 서비스 응답 속도가 일정하게 유지되어야 한다면 G1GC가 가장 강력한 무기가 됩니다.

자바의 역사 속에는 다양한 GC들이 존재합니다. 각 GC는 시대적 배경이나 특정 목적(속도, 처리량, 메모리 절약 등)에 따라 탄생했습니다. 

# 1. 주요 GC 5종 세트 비교 
GC 종류       |    특징                       |   핵심 목표                |    비고         
Parallel GC  | 여러개의 쓰레드로 병렬 처리         | 처리량(Throughput) 극대화   | Java 8의 기본 GC
CMS GC       | 애플리케이션과 GC 쓰레드 동시 실행   | 낮은 지연 시간(Low Latency) | Java 14에서 삭제됨
G1 GC        | 힙을 리전(Region)으로 나눠 관리    | 예측 가능한 중단 시간         | Java 9 이후 기본 GC
ZGC          | 컬러드 포인터, 부하 분산 기술 사용   | 초저지연 (STW < 1ms)       | 대규모 힙(TB 단위)에 유리
Shenandoah   | Red Hat에서 개발, 단계별 동시 압축 | 초저지연                   | ZGC와 유사한 목적

# 2. 상세 설명
1) Parallel GC (병렬 GC)
    * 방식: Minor GC를 멀티 쓰레드로 수행합니다.
    * 장점: 개별 청소 시간을 좀 걸릴 수 있어도, 전체적인 작업 처리량은 가장 좋습니다.
    * 용도: 사용자 응답 속도보다 '배치 작업'처럼 대량의 데이터를 빨리 처리해야 하는 서버에 적합니다.

2) CMS GC (Concurrent Mark Sweep)
    * 방식: 애플리케이션이 돌아가는 와중에 GC가 몰래몰래 쓰레기를 표시(Mark)하고 지웁니다. (Sweep)
    * 한계: 메모리 파편화(구멍 숭숭 뚫림)를 해결하는 '압축(Compaction)' 기능이 없어 결국 나중에 아주 긴 정지 시간이 발생합니다. 이 문제로 인해 G1GC에 자리를 넘겨주고 퇴출되었습니다. 

3) ZGC & Shenandoah (차세대 GC)
    * 방식:  "멈춤 없는 GC"를 지향합니다. 힙 메모리가 8MB든 16TB든 상관없이 '정지 시간을 1ms 이하'로 유지하는 게 목표입니다.
    * 장점: 실시간성이 중요한 금융 시스템이나 대규모 게임 서버 등에 사용됩니다. 

# 3. 그 외의 독특한 GC들
    * Serial GC: 쓰레드 딱 1개로만 청소합니다. CPU 코어가 1개뿐이던 시절의 유물이나, 메모리가 극도로 적은 임베디드 기기에서 가끔 사용됩니다. 
    * Epsilon GC: '아무것도 안 하는 GC'입니다. 메모리가 꽉 차면 그냥 에러(OOM)를 내고 죽습니다. 
        * 왜 쓸까? GC 오버헤드가 아예 없는 상태에서 성능 테스트를 하고 싶을 때나, 메모리 관리가 필요없는 아주 짧은 작업용으로 쓰입니다. 

# 4. 요약: 백엔드 개발자의 선택
    * 보통 G1GC를 그대로 쓰는 게 가장 속 편하고 성능도 좋다.
    * 만약 '우리 서비스는 응답 속도가 무조건 10ms 이내여야 해!'라고 한다면 'ZGC'를 검토
